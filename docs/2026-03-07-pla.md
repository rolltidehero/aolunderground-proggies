ARCHITECTURE OVERVIEW
---------------------
Three host↔guest communication channels, ZERO network:

  1. QMP (QEMU Machine Protocol)
     - Host socket: /tmp/vm-decompiler-qmp.sock
     - Purpose: VM lifecycle, screendumps, keyboard/mouse input
     - Direction: Host -> QEMU hypervisor (not guest OS)

  2. QGA (QEMU Guest Agent)
     - Host socket: /tmp/vm-decompiler-qga.sock
     - Guest transport: virtio-serial port "org.qemu.guest_agent.0"
     - Purpose: File transfer (push EXEs in, pull decompiled output out),
       ad-hoc command execution, diagnostics
     - Direction: Host <-> Guest OS (via virtio-serial, no network)
     - Limitations: Base64 encoded, 48MB per-read limit, adequate for
       our file sizes (50KB-2MB), not a bulk-transfer substrate
     - Security: Safer than virtiofs but NOT a security boundary.
       Red Hat warns it is only safe for trusted guests. Acceptable
       for our use case (we control both sides, no network, no remote
       attackers). Harden with RPC allow-lists in runtime mode.

  3. C2 (Custom virtio-serial agent)
     - Host socket: /tmp/vm-decompiler-c2.sock
     - Guest device: \\.\Global\c2channel
     - Purpose: Custom commands to agent.py (decompile, status, shell)
     - Direction: Host <-> Guest OS (via virtio-serial, no network)

File transfer strategy:
  - share.img (USB disk image): Bootstrap-only. Used ONCE in Phase 1
    to cold-stage files. Cannot be safely host-mounted while guest
    owns the filesystem. Once QGA is working, share.img is retired
    from active use.
  - QGA guest-file-*: Primary live file transfer for all subsequent
    operations. 1MB chunking built in from the start.

Security constraints:
  - VM has ZERO network access (-nic none)
  - No virtiofs on this VM (untrusted executables = host exposure risk)
  - QEMU process runs as unprivileged user with AppArmor profile
  - Snapshots restore clean state every 50 EXEs or on failure
  - Output files sanitized (filename validation, size limits, quarantine dir)
  - QGA RPC allow-list in runtime mode (Phase 7)

--- T1.1: UPDATE THE QEMU LAUNCH SCRIPT ---

The launch script must define THREE virtio-serial channels.
File: ~/vm/launch-decompiler.sh

#!/bin/bash
VMDIR=~/vm
DISK=$VMDIR/win10-analysis.qcow2
SHARE=$VMDIR/share.img
VIRTIO_ISO=$VMDIR/virtio-win.iso  # or wherever it is

qemu-system-i386 \
  -name vm-decompiler-no-internet \
  -machine type=pc,accel=kvm \
  -m 4096 \
  -cpu host \
  -smp 2 \
  -drive file=$DISK,format=qcow2,if=virtio \
  -drive file=$SHARE,format=raw,if=none,id=usb-share \
  -device usb-storage,drive=usb-share \
  -nic none \
  -display none \
  -vnc :1 \
  \
  # --- Communication Channels --- \
  \
  # Virtio-serial bus (one bus, multiple ports) \
  -device virtio-serial-pci,id=virtio-serial0 \
  \
  # Channel 1: QMP (VM control - not on virtio-serial, separate) \
  -qmp unix:/tmp/vm-decompiler-qmp.sock,server,nowait \
  \
  # Channel 2: Custom C2 agent channel \
  -chardev socket,id=c2char,path=/tmp/vm-decompiler-c2.sock,server,nowait \
  -device virtserialport,bus=virtio-serial0.0,chardev=c2char,name=c2channel \
  \
  # Channel 3: QGA channel \
  -chardev socket,id=qgachar,path=/tmp/vm-decompiler-qga.sock,server,nowait \
  -device virtserialport,bus=virtio-serial0.0,chardev=qgachar,name=org.qemu.guest_agent.0 \
  \
  -daemonize \
  -pidfile /tmp/vm-decompiler.pid

NOTE ON CHANNEL NAMES:
  - "c2channel" matches what agent.py opens as \\.\Global\c2channel
  - "org.qemu.guest_agent.0" is the standard name the Windows QGA
    service searches for. Do not rename without confirming your QGA
    build supports alternatives.

NOTE ON ARCHITECTURE:
  - qemu-system-i386 because guest is Windows 10 x86 (32-bit)
  - Use qemu-system-x86_64 ONLY if guest is 64-bit

--- T2.2: INSTALL VIOSERIAL DRIVER (via QMP keyboard) ---

  NOTE: The ACCESS_DENIED error on \\.\Global\c2channel suggests the
  vioserial driver is likely ALREADY loaded (the device path exists).
  This step is a safety measure. If the driver is already present,
  pnputil will say so and continue harmlessly.

  The share.img should appear as a USB drive in Windows (likely D: or E:).
  Determine the drive letter by taking a screendump after opening Explorer,
  or just try D: first.

  Using input_controller.py (or raw QMP send-key commands):

  Step 1: Open Run dialog
    QMP: send Win+R key combo
    Wait 2 seconds

  Step 2: Type and execute vioserial install
    QMP: type string: pnputil /add-driver D:\staging\vioserial\vioser.inf /install
    QMP: send Enter key
    Wait 15 seconds for PnP enumeration

  Step 3: Take screendump to verify result
    QMP screendump /tmp/vioserial-install.ppm

  If pnputil shows an error about wrong drive letter, try E:\ instead.

--- T2.3: INSTALL QGA (via QMP keyboard) ---

  Step 1: Open Run dialog
    QMP: send Win+R key combo
    Wait 2 seconds

  Step 2: Type and execute QGA install
    QMP: type string: msiexec /i D:\staging\qga\qemu-ga-i386.msi /quiet /norestart
    QMP: send Enter key
    Wait 30 seconds for MSI to complete

  Step 3: Take screendump to verify
    QMP screendump /tmp/qga-install.ppm

  Alternative (if using full guest-tools installer instead):
    msiexec /i D:\staging\qga\virtio-win-gt-x86.msi /quiet /norestart
    This installs ALL VirtIO drivers + QGA. Simpler but installs more
    than strictly needed.

--- T2.4: VERIFY QGA — FULL TEST SUITE ---

  QGA is only "working for this workflow" when ALL FIVE tests pass.

  Connect to QGA socket:
    All commands below are JSON sent to /tmp/vm-decompiler-qga.sock
    Use: echo '<json>' | socat - UNIX:/tmp/vm-decompiler-qga.sock

  IMPORTANT: QGA requires a sync handshake first:
    {"execute":"guest-sync-delimited","arguments":{"id":12345}}
    Expected response: {"return": 12345}

  Test 1: Ping
    {"execute":"guest-ping"}
    Expected: {"return":{}}
    This confirms: QGA service is running, virtio-serial transport works.

  Test 2: Command execution
    {"execute":"guest-exec","arguments":{"path":"C:\\Windows\\System32\\cmd.exe","arg":["/c","whoami"],"capture-output":true}}
    Expected: {"return":{"pid":<number>}}
    Then poll:
    {"execute":"guest-exec-status","arguments":{"pid":<number>}}
    Expected: {"return":{"exited":true,"exitcode":0,"out-data":"<base64>"}}
    Decode out-data, should contain "lab" or "nt authority\system"

  Test 3: File write
    {"execute":"guest-file-open","arguments":{"path":"C:\\qga_test.txt","mode":"w"}}
    Expected: {"return":<handle>}
    {"execute":"guest-file-write","arguments":{"handle":<handle>,"buf-b64":"SGVsbG8gZnJvbSBob3N0IQ=="}}
    (That's "Hello from host!" in base64)
    Expected: {"return":{"count":16,"eof":false}}
    {"execute":"guest-file-close","arguments":{"handle":<handle>}}
    Expected: {"return":{}}

  Test 4: File read-back
    {"execute":"guest-file-open","arguments":{"path":"C:\\qga_test.txt","mode":"r"}}
    Expected: {"return":<handle>}
    {"execute":"guest-file-read","arguments":{"handle":<handle>,"count":1024}}
    Expected: {"return":{"count":16,"buf-b64":"SGVsbG8gZnJvbSBob3N0IQ==","eof":true}}
    Decode and verify content matches "Hello from host!"
    {"execute":"guest-file-close","arguments":{"handle":<handle>}}

  Test 5: Cleanup
    {"execute":"guest-exec","arguments":{"path":"C:\\Windows\\System32\\cmd.exe","arg":["/c","del","C:\\qga_test.txt"],"capture-output":true}}
    Poll for completion.

  IF ANY TEST FAILS:
    - Test 1 fails: QGA service not running or transport broken.
      Check: QGA service status via QMP keyboard:
        Win+R -> cmd /c "sc query \"QEMU Guest Agent\"" > C:\qga_status.txt
      Then screendump or try reading the file.
    - Test 2 fails: QGA runs but cannot exec. Check QGA version/build.
    - Test 3/4 fails: File I/O broken. Check disk permissions, antivirus.
    - Investigate and fix before proceeding.

--- T2.5: THIS IS THE LAST PLANNED SHUTDOWN FOR BOOTSTRAP STAGING ---

  share.img is no longer needed for file transfer. All subsequent file

PHASE 3: DIAGNOSE AND FIX CUSTOM VIRTIO-SERIAL AGENT
=============================================================================
Purpose: Get agent.py working on \\.\Global\c2channel so we have our
custom C2 channel for decompile commands and GUI automation.

The known symptom: agent.py gets "Access is denied" (error 5) when
calling CreateFile("\\.\Global\c2channel").

--- T3.1: DIAGNOSTIC SEQUENCE (in this order) ---

  All diagnostics run via QGA guest-exec from the host.

  CHECK 1: Is the vioserial DRIVER loaded?
    QGA guest-exec: cmd /c "pnputil /enum-devices /class {4D36E978-E325-11CE-BFC1-08002BE10318}" > C:\diag_driver.txt
    QGA guest-file-read: C:\diag_driver.txt
    Look for: "VirtIO Serial Driver" or "vioserial" in output.
    Also try: cmd /c "driverquery /v | findstr -i virt" > C:\diag_driver2.txt

  CHECK 2: Does the DEVICE PATH exist?
    QGA guest-exec: cmd /c "dir \\.\Global\" > C:\diag_path.txt 2>&1
    Note: dir on \\.\Global\ may not work. Alternative:
    QGA guest-exec: powershell -c "Get-WmiObject Win32_PnPEntity | Where-Object {$_.Name -like '*VirtIO*' -or $_.Name -like '*Serial*'} | Select-Object Name,Status | Out-File C:\diag_devices.txt"
    QGA guest-file-read: C:\diag_devices.txt

  CHECK 3: Is the CHANNEL NAME correct?
    The QEMU launch script sets name=c2channel on the virtserialport.
    In Windows, this appears as \\.\Global\c2channel (the name= value).
    Verify agent.py uses exactly: \\.\Global\c2channel (not \\.\c2channel,
    not \\.\COM3, not any other path).
    QGA guest-exec: cmd /c "type C:\path\to\agent.py | findstr c2channel" > C:\diag_agent_path.txt
    QGA guest-file-read: C:\diag_agent_path.txt

  CHECK 4: Is another PROCESS already holding the port?
    Virtio-serial ports allow only one open handle at a time.
    If a previous agent.py instance crashed without closing the handle,
    or if QGA or another process grabbed it, new opens fail.
    QGA guest-exec: cmd /c "tasklist | findstr -i python" > C:\diag_procs.txt
    QGA guest-file-read: C:\diag_procs.txt
    If python.exe is running: kill it.
    QGA guest-exec: cmd /c "taskkill /f /im python.exe"

  CHECK 5: Is it a PERMISSION / SECURITY CONTEXT issue?
    The agent may need to run as SYSTEM, not as the "lab" user.
    QGA guest-exec: cmd /c "whoami" > C:\diag_whoami.txt
    QGA guest-file-read: C:\diag_whoami.txt
    If agent.py was started as "lab" user via Task Scheduler:
      Fix: Change the scheduled task to run as SYSTEM:
      QGA guest-exec: schtasks /change /tn "AgentStartup" /ru SYSTEM
      Or create a new task:
      QGA guest-exec: schtasks /create /tn "AgentC2" /tr "python C:\agent\agent.py" /sc onlogon /ru SYSTEM /rl HIGHEST /f

  OPTIONAL CHECK 6: Event Viewer for PnP errors
    QGA guest-exec: powershell -c "Get-WinEvent -LogName System -MaxEvents 30 | Where-Object {$_.Message -like '*virtio*' -or $_.Message -like '*vioserial*' -or $_.Message -like '*serial*'} | Format-List TimeCreated,Message | Out-File C:\diag_events.txt"
    QGA guest-file-read: C:\diag_events.txt
    Look for driver load failures, device enumeration errors.

--- T3.2: APPLY FIX ---

  Most likely resolution order (based on evidence):
    1. Kill stale python.exe (Check 4) -> retry
    2. Run as SYSTEM (Check 5) -> retry
    3. Install vioserial driver (Check 1, if missing) -> reboot -> retry
    4. Fix channel name mismatch (Check 3) -> fix agent.py -> retry

  After fix, test the agent:
    From host, connect to C2 socket and send a test command:
      echo "STATUS" | socat - UNIX:/tmp/vm-decompiler-c2.sock
    Expected: agent.py responds with status JSON.

--- T3.3: VERIFY AGENT BASIC FUNCTIONS ---

  Test: echo "SHELL whoami" | socat - UNIX:/tmp/vm-decompiler-c2.sock
  Test: echo "SHELL dir C:\\" | socat - UNIX:/tmp/vm-decompiler-c2.sock
  Test: echo "STATUS" | socat - UNIX:/tmp/vm-decompiler-c2.sock

  All should return valid responses. If they do, agent.py is working.

PHASE 7: BATCH PIPELINE
=============================================================================
Purpose: Process thousands of EXEs automatically with safety controls.

PLUGIN API (CONFIRMED 2026-03-08):
  Official SDK: https://www.vb-decompiler.org/plugins_sdk.htm
  Export signatures: void __stdcall Func(HWND mainWnd, HWND richEditWnd, char* buffer, void* engine)
  Engine pointer is 4th param of PluginLoad (NOT 1st — 1st is main window HWND)
  Engine callback: wchar_t* __stdcall (int vlType, int vlNumber, int vlFnNumber, char* vlNewValue)
  Plugin activation: WM_COMMAND ID 60 to TfrmMain (after file is loaded+decompiled)

--- T7.1: HOST-SIDE BATCH ORCHESTRATOR ---

  File: batch_decompile.py (runs on Linux host)

  Configuration:
    INPUT_DIR = "/path/to/exe/collection/"       # thousands of EXEs
    OUTPUT_DIR = "/path/to/decompiled/output/"    # quarantine directory
    QUARANTINE_DIR = "/path/to/quarantine/"       # sanitized output
    QGA_SOCK = "/tmp/vm-decompiler-qga.sock"
    C2_SOCK = "/tmp/vm-decompiler-c2.sock"
    QMP_SOCK = "/tmp/vm-decompiler-qmp.sock"

    TIMEOUT_PER_FILE = 180                        # seconds
    MAX_OUTPUT_SIZE = 50 * 1024 * 1024            # 50MB per file output
    FFMPEG_ENABLED = True

  Main loop pseudocode:

    exe_list = sorted(glob(INPUT_DIR + "/*.exe"))
    processed = load_progress("progress.json")  # resume support

    for batch_start in range(0, len(exe_list), BATCH_SIZE):
        batch = exe_list[batch_start : batch_start + BATCH_SIZE]

        # --- Restore clean snapshot ---
        qmp_command("system_powerdown")
        wait_for_shutdown(timeout=60)
        run("qemu-img snapshot -a production-clean win10-analysis.qcow2")

        # --- Boot VM ---
        run("~/vm/launch-decompiler.sh")
        wait_for_boot(timeout=120)  # poll QGA guest-ping

        # --- Start video recording ---
        if FFMPEG_ENABLED:
            ffmpeg_proc = start_ffmpeg_recording(batch_start)
            # ffmpeg connects to VNC :1 and records to MP4
            # ffmpeg -f vnc -i localhost:5901 -c:v libx264 -crf 23
            #        batch_{batch_start}.mp4

        # --- Wait for agent ---
        wait_for_agent(C2_SOCK, timeout=60)  # poll STATUS command

        # --- Process each EXE ---
        for exe_path in batch:
            exe_name = os.path.basename(exe_path)
            safe_name = sanitize_filename(exe_name)  # strip weird chars

            if safe_name in processed:
                continue

            try:
                # 1. Push EXE to guest via QGA
                push_file(QGA_SOCK, exe_path, f"C:\\work\\input\\{safe_name}")

                # 2. Send decompile command via C2
                send_c2(C2_SOCK, f"DECOMPILE C:\\work\\input\\{safe_name} C:\\work\\output\\{safe_name}")

                # 3. Wait for completion (poll C2 for status)
                result = wait_for_completion(C2_SOCK, timeout=TIMEOUT_PER_FILE)

                if result == "SUCCESS":
                    # 4. Pull output files via QGA
                    output_files = list_guest_dir(C2_SOCK, f"C:\\work\\output\\{safe_name}")
                    for f in output_files:
                        # Validate filename
                        if not is_safe_filename(f):
                            log_warning(f"Skipping unsafe filename: {f}")
                            continue
                        guest_path = f"C:\\work\\output\\{safe_name}\\{f}"
                        local_path = os.path.join(QUARANTINE_DIR, safe_name, f)
                        os.makedirs(os.path.dirname(local_path), exist_ok=True)
                        pull_file(QGA_SOCK, guest_path, local_path)
                        # Validate size
                        if os.path.getsize(local_path) > MAX_OUTPUT_SIZE:
                            log_warning(f"Output too large: {local_path}")
                            os.remove(local_path)

                    processed[safe_name] = "success"
                else:
                    processed[safe_name] = f"failed: {result}"
                    log_error(f"Failed: {exe_name}: {result}")

                # 5. Clean up guest working dirs
                send_c2(C2_SOCK, f"SHELL del /q C:\\work\\input\\{safe_name}")
                send_c2(C2_SOCK, f"SHELL rmdir /s /q C:\\work\\output\\{safe_name}")

            except TimeoutError:
                processed[safe_name] = "timeout"
                log_error(f"Timeout: {exe_name}")
                # Force-kill VB Decompiler in guest
                send_c2(C2_SOCK, "SHELL taskkill /f /im VBDecompiler.exe")
                time.sleep(5)

            except Exception as e:
                processed[safe_name] = f"error: {str(e)}"
                log_error(f"Error: {exe_name}: {e}")
                # If agent is unresponsive, break to snapshot restore
                if not agent_responsive(C2_SOCK):
                    log_error("Agent unresponsive, breaking batch")
                    break

            # Save progress after each file
            save_progress("progress.json", processed)

        # --- Stop video recording ---
        if FFMPEG_ENABLED:
            stop_ffmpeg_recording(ffmpeg_proc)

    # --- Final report ---
    generate_report(processed)

--- T7.2: FILE TRANSFER HELPER FUNCTIONS ---

  push_file(qga_sock, local_path, guest_path):
      with open(local_path, "rb") as f:
          data = f.read()
      handle = qga_call("guest-file-open", path=guest_path, mode="w")
      CHUNK_SIZE = 1024 * 1024  # 1MB
      for offset in range(0, len(data), CHUNK_SIZE):
          chunk = data[offset:offset+CHUNK_SIZE]
          b64 = base64.b64encode(chunk).decode()
          qga_call("guest-file-write", handle=handle, **{"buf-b64": b64})
      qga_call("guest-file-close", handle=handle)

  pull_file(qga_sock, guest_path, local_path):
      handle = qga_call("guest-file-open", path=guest_path, mode="r")
      with open(local_path, "wb") as f:
          while True:
              result = qga_call("guest-file-read", handle=handle, count=1048576)
              if result["count"] == 0:
                  break
              data = base64.b64decode(result["buf-b64"])
              f.write(data)
              if result.get("eof", False):
                  break
      qga_call("guest-file-close", handle=handle)

--- T7.3: FILENAME SANITIZATION ---

  def sanitize_filename(name):
      # Allow only alphanumeric, dash, underscore, dot
      safe = re.sub(r'[^a-zA-Z0-9._-]', '_', name)
      # Prevent directory traversal
      safe = safe.replace('..', '_')
      # Truncate
      if len(safe) > 200:
          safe = safe[:200]
      return safe

  def is_safe_filename(name):
      if '..' in name or '/' in name or '\\' in name:
          return False
      if len(name) > 255:
          return False
      return True

WHAT IS EXPLICITLY NOT IN THIS PLAN (AND WHY)
=============================================================================

- virtiofs: Excluded for VM2. Untrusted EXEs could exploit FUSE
  passthrough to access host filesystem. Non-negotiable.

- 9p filesystem: No Windows guest driver exists.

- DCOM/VSS permission changes: Irrelevant to virtio-serial device
  access. DCOM controls COM server activation; our issue is a kernel
  device object DACL. Different subsystem entirely.

- guest-fsfreeze during decompilation: Would halt all disk I/O in the
  guest, breaking VB Decompiler. Only use AFTER decompilation for
  optional live snapshot optimization:
    decompile -> pull output -> freeze -> snapshot -> thaw

- Blocking guest-exec during provisioning: Phases 2-6 require
  guest-exec. Only restrict in Phase 7 runtime mode.
